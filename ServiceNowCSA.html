📌 1. ServiceNow Fundamentals

▼

What is ServiceNow?

ServiceNow is a cloud-based platform used by companies to manage IT services, automate workflows, and streamline business processes.

Companies use ServiceNow for:
  • IT Service Management (ITSM)
  • Employee workflows
  • Asset management
  • Incident and problem tracking
  • Automation of business processes

1. Instance

An instance is your ServiceNow environment — a dedicated server with its own database, applications, and configurations.

Example URL: https://devXXXXX.service-now.com

Instance TypePurpose
Developer Instance (PDI)Personal practice (free)
Test InstanceTesting configurations
Production InstanceLive usage by real users

2. Tables

A table stores data — similar to a spreadsheet or database table.

Table LabelTable NamePurpose
IncidentincidentStores incident tickets
ProblemproblemStores problems
Change Requestchange_requestStores changes
Usersys_userStores users
Groupsys_user_groupStores groups

3. Records

A record = one row in a table.

Example: INC0010001, INC0010002 are records in the incident table.

4. Forms

A form is the detailed view of a single record — shows all fields like Number, Caller, State, Priority.

5. Lists

A list shows multiple records in a table/grid format. Lists allow Filtering, Sorting, Searching.

6. Applications & Modules

Application = category of features. Module = specific page/link inside an application.

Example: Application → Incident | Modules → Create New, Open, All
Q: Difference between Table, Record, Form, and List?
Table → stores data | Record → single row | Form → detailed view of one record | List → multiple records in table format

📌 2. Tables, Fields & Dictionary

▼

Fields (Columns)

Field TypeExample
StringShort description
IntegerPriority
ReferenceCaller (links to sys_user)
Date/TimeCreated date
ChoiceState (New, In Progress, Closed)
BooleanActive (true/false)

Reference Field IMPORTANT

A reference field creates a relationship between two tables.

Example: Caller field on Incident → references sys_user table.

Dictionary IMPORTANT

The Dictionary stores metadata of fields — type, max length, mandatory status, default value.

Navigation: System Definition → Dictionary
Q: What is a Dictionary in ServiceNow?
Dictionary defines properties of fields in a table like type, length, default value, and behavior.

Table Extension (Parent–Child) MOST IMPORTANT

Task (Parent Table)
   ├── Incident
   ├── Problem
   └── Change Request

Task = base table. Child tables inherit all Task fields automatically. Incident then adds its own fields (Impact, Urgency, Priority).

Verify: System Definition → Tables → Open incident → Check "Extends Table = Task"
Q: Why is the Task table important?
Task is the base table providing common fields and structure for all task-based applications like Incident, Problem, and Change. This avoids duplicate fields through inheritance.

📌 3. Forms & List Customization

▼

Form Layout Customization

Steps: Open Incident record → Right-click header → Configure → Form Layout → Move fields → Save

List Layout Customization

Steps: Go to Incident → All → Right-click column header → Configure → List Layout → Add/remove columns → Save

Making Fields Mandatory / Read-only / Hidden

Dictionary (Static)

Always mandatory. No conditions.

UI Policy (Dynamic) ⭐

Mandatory only when condition is met.

Q: When to use UI Policy vs Dictionary?
Dictionary → field is ALWAYS mandatory (static). UI Policy → field becomes mandatory only when a condition is met (dynamic).

📌 4. UI Policies

▼

What is a UI Policy?

UI Policy dynamically controls form field behavior without code. You can make fields: Mandatory, Read-only, Visible/Hidden.

Example: If State = Closed → Make "Close Notes" mandatory.

Types of UI Policies

TypeWhere UsedFocus?
System UI PolicyIncident, Problem, Change forms✅ YES (Main)
Catalog UI PolicyService Catalog variablesLater
Mobile UI PolicyMobile app❌ Skip
Debug UI PolicyTroubleshooting❌ Skip

How to Create a UI Policy

  1. Navigate to System UI → UI Policies
  2. Click New
  3. Table → Incident
  4. Condition → State = Closed
  5. Submit
  6. Add UI Policy Action: Field → Close Notes, Mandatory → true
  7. Save
Q: UI Policy vs Client Script?
UI Policy → no coding, simple conditions. Client Script → uses JavaScript, more powerful.

📌 6. ACL (Access Control) CRITICAL

▼

What is ACL?

ACL = Access Control List. Controls who can read, write, create, or delete data.

OperationMeaning
readView data
writeEdit data
createCreate new records
deleteDelete records

How ACL Works

User Request→ Role Check→ ACL Check→ Granted / Denied

ACL Levels

  • Table level → controls entire table
  • Field level → controls specific field

ACL Can Have

  • Condition (no code) — e.g., State = Active
  • Script — e.g., answer = gs.hasRole('admin');
Create ACL: Access Control (ACL) → New → Type: Record → Operation: read → Table: Incident → Requires role: itil → Submit

Role

Gives permission. Assigned to user.

ACL

Enforces permission. Defined on table/field.

Q: What is ACL?
ACL controls access to records and fields based on roles, conditions, and scripts.
Q: Can UI Policy be used for security?
❌ No. UI Policy is only for UI behavior. Security must use ACL.

📌 7. Service Catalog HIGH PRIORITY

▼

What is Service Catalog?

Service Catalog creates request forms for users to request services or items.

Examples: Laptop request, Leave request, Software installation.

Key Components

ComponentMeaning
Catalog ItemThe request form users fill
VariablesInput fields in the form
Record ProducerCreates record in a table
Order GuideBundles multiple items

Variable Types

TypeExample
Single Line TextEmployee Name
Select BoxLaptop Type (Dell/HP/Lenovo)
Yes/NoNeed Accessories?
ReferenceUser lookup

How Catalog Works

User→ Opens Catalog→ Fills Form→ Submits→ Request Created
Q: Catalog Item vs Record Producer?
Catalog Item → request services (creates request + requested items). Record Producer → directly creates record in a specific table.
Q: UI Policy vs Catalog UI Policy?
UI Policy → works on table fields. Catalog UI Policy → works on catalog variables.

📌 8. Notifications & Events

▼

What is a Notification?

Sends emails automatically based on actions/events.

Event/Trigger→ Notification Rule→ Email Sent
Create: Notifications → New → Table: Incident → When: Inserted → Who: Caller → Subject & Body → Save
Q: What is a Notification?
Used to send automated emails when specific events or conditions occur (e.g., incident created, assigned, breached).

📌 9. Import Sets & Data Loading

▼

What is Import Set?

Used to load external data (Excel/CSV) into ServiceNow.

Excel/CSV→ Import Set Table→ Transform Map→ Target Table

Key Concepts

  • Import Set Table → temporary staging table
  • Transform Map → maps source columns to target fields
  • Coalesce → decides INSERT new or UPDATE existing
Steps: Load Data → Upload file → Create Transform Map → Map fields → Run Transform
Q: What is Coalesce?
Coalesce decides whether to insert a new record or update an existing one. If a matching record is found, it updates; otherwise it inserts.

📌 10. Reports & Dashboards

▼

What is a Report?

Used to visualize and analyze data. Types: Bar, Pie, Line, List, Donut charts.

Create: Reports → Create New → Table: Incident → Type: Bar chart → Group by: Priority → Run

Dashboards

A Dashboard = collection of multiple reports/widgets on a single page.

Q: Why use Reports?
To analyze and visualize data — e.g., showing incidents by priority, tracking trends over time.

📌 11. SLA (Service Level Agreement)

▼

What is SLA?

SLA defines time limits to complete tasks.

Example: P1 incidents must be resolved within 4 hours.
Incident Created→ SLA Starts→ Timer Runs→ Breach if Delayed

Key Concepts

  • Start Condition → when SLA clock starts
  • Stop Condition → when SLA clock stops
  • Breach → task NOT completed in time
Q: What is SLA Breach?
An SLA breach occurs when a task is not completed within the defined time limit.

📌 12. Additional CSA Topics

▼

Update Sets

Used to move configurations from one instance to another (Dev → Test → Prod).

Create Update Set → Make changes → Export (XML) → Import in target → Preview → Commit
Q: What is an Update Set?
Used to move configurations and customizations between instances (like dev to production).

Data Policy

Similar to UI Policy but works on server side also.

UI Policy → enforces rules only on the form (client).
Data Policy → enforces rules even through imports, web services, APIs (server).
Q: UI Policy vs Data Policy?
UI Policy → client-side form behavior. Data Policy → enforces data rules even through imports, APIs, scripts.

Business Rule (Basic for CSA)

Executes server-side logic when records are inserted, updated, queried, or deleted.

Types: before, after, async, display. Example: Priority = 1 → auto-assign group.
Q: Client Script vs Business Rule?
Client Script → runs on browser (client-side). Business Rule → runs on server (backend).

Knowledge Management

Create/manage knowledge articles (FAQs, how-to guides) to help users find answers without tickets.

CMDB

CMDB stores info about Configuration Items (CIs) — hardware, software, services and their relationships.

Q: What is CMDB?
CMDB is a database that stores information about Configuration Items (CIs) like servers, applications, and network devices, along with their relationships.

ITSM Key Processes

ProcessPurpose
Incident ManagementRestore service quickly
Problem ManagementFind root cause
Change ManagementImplement permanent solution safely
Request FulfillmentHandle user service requests
Q: Difference between Incident, Problem, and Change?
Incident → fix immediate issue. Problem → find root cause. Change → implement permanent solution.

📌 13. Complete Interview Questions & Answers

▼

Basic Questions

Q1: What is ServiceNow?
ServiceNow is a cloud-based platform used for IT service management to automate workflows and manage services like incidents, problems, and requests.
Q2: What is an Instance?
An instance is a single ServiceNow environment with its own database, applications, and configurations.
Q3: What is a Table?
A table is a collection of records used to store data.
Q4: What is a Reference Field?
A reference field links one table to another. Example: Caller → sys_user.
Q5: What is a Module?
A module is a link in the application navigator used to access records or features.

Admin Questions

Q6: What is UI Policy?
UI Policy dynamically controls field behavior (mandatory, read-only, visibility) based on conditions without scripting.
Q7: What is ACL?
ACL controls access to data by defining who can read, write, create, or delete records based on roles, conditions, and scripts.
Q8: Role vs ACL?
Role → gives permission (assigned to user). ACL → enforces permission (defined on table/field).
Q9: What is Service Catalog?
Service Catalog creates request forms for users to request services or items.
Q10: What is a Notification?
Sends automated emails when certain conditions are met.
Q11: What is Import Set?
Used to load external data (Excel/CSV) into ServiceNow.
Q12: What is Transform Map?
Maps data from import set table to target table.
Q13: What is SLA?
SLA defines the time limit to complete a task. Breach = task not completed in time.
Q14: What is Update Set?
Used to move configurations from one instance to another.
Q15: What is Table Extension?
One table inherits fields from another. Example: Incident extends Task, getting all Task fields.

Comparison Questions

Q16: UI Policy vs Dictionary?
Dictionary → static (always applies). UI Policy → dynamic (condition-based).
Q17: UI Policy vs Data Policy?
UI Policy → client-side only (forms). Data Policy → works on server side too (imports, APIs).
Q18: Client Script vs Business Rule?
Client Script → browser, UI-related. Business Rule → server, backend logic.
Q19: Catalog Item vs Record Producer?
Catalog Item → request services. Record Producer → creates records in a specific table.
Q20: Incident vs Problem vs Change?
Incident → fix immediate issue. Problem → find root cause. Change → permanent solution.

Tricky Questions

Q: Can UI Policy be used for security?
❌ No. UI Policy is only for UI behavior. Use ACL for security.
Q: Can a user access data without a role?
❌ Generally no. No role = no access.
Q: What happens if no ACL is defined?
System applies default access rules (usually restrictive).

📌 14. Scenario-Based Questions

▼
Scenario 1: If Incident is Closed → Close Notes must be mandatory
✅ Solution: Use UI Policy — Condition: State = Closed → Action: Close Notes mandatory = true
Scenario 2: Only ITIL users should access incidents
✅ Solution: Use ACL with itil role on Incident table (read operation)
Scenario 3: When incident is created → send email
✅ Solution: Use Notification — When: Inserted, Table: Incident
Scenario 4: Load 1000 employee records from Excel
✅ Solution: Use Import Set + Transform Map
Scenario 5: Show incidents based on priority
✅ Solution: Use Report — Bar chart grouped by Priority
Scenario 6: Hide a field only in certain condition
✅ Solution: Use UI Policy — Set field visible = false based on condition
Scenario 7: Move customizations from Dev to Prod
✅ Solution: Use Update Sets — Export from Dev, Import to Prod, Preview, Commit
Scenario 8: Only assigned user should edit their incident
✅ Solution: Use ACL (write) with condition: Assigned to = current user
Scenario 9: Employees need to request laptops through a portal
✅ Solution: Create Catalog Item with variables (name, laptop type, accessories) in Service Catalog
Scenario 10: Ensure field is mandatory even when data comes via API
✅ Solution: Use Data Policy (not UI Policy, since UI Policy only works on forms)
