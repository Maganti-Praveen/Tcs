What is DBMS?
Database Management System — fundamentals
🗄️

DBMS Overview

▼

A Database Management System (DBMS) is software that manages the creation, storage, retrieval and security of data in a structured way. Instead of saving data in flat files (like CSVs), a DBMS organises data into tables, enforces relationships, and provides a query language (SQL) to interact with it.

File System vs DBMS

File systems store raw data with no structure. DBMS adds querying, relationships, security, and concurrency — critical for fintech like CreditMantri handling crores of transactions.

Key Functions

Data Definition (DDL) · Data Manipulation (DML) · Data Control (DCL) · Transaction Control (TCL)

Popular DBMS

MySQL, PostgreSQL (Relational) · MongoDB (Document) · Redis (Key-Value) · Cassandra (Wide-Column) · Neo4j (Graph)

🔑
Keys — Primary & Foreign
🗝️

Types of Keys with Diagrams

▼

Keys uniquely identify rows and enforce relationships between tables. Every well-designed relational database relies on them.

🔑 Primary Key (PK)

Uniquely identifies each row. Cannot be NULL. Only ONE per table. Example: user_id in users table.

🔗 Foreign Key (FK)

A column that references a PK in another table. Enforces referential integrity. Prevents orphan records.

🪪 Candidate Key

Any column(s) that could be a PK. A table can have multiple candidate keys; the one chosen becomes the PK.

🔄 Composite Key

A PK made of two or more columns together. Common in many-to-many junction tables. E.g., (user_id + lender) as a combined key.

🪄 Surrogate Key

An artificial PK created by the system (AUTO_INCREMENT). Has no business meaning. Preferred over natural keys for stability.

🌍 Natural Key

A PK derived from real-world data (e.g., PAN number, Aadhaar). Can change over time — generally less preferred.

Joins — Visual Guide
🔗

All Join Types with Venn Diagrams

▼

A JOIN combines rows from two or more tables based on a related column. The type of join determines which rows are included when there is no match.

CROSS JOIN

Returns Cartesian product — every row of A with every row of B. 20 users × 25 loans = 500 rows. Rarely used intentionally.

SELF JOIN

A table joined with itself using aliases. Used for hierarchical data (employee → manager). e.g. users u1 JOIN users u2 ON u1.referrer_id = u2.user_id

ACID Properties
🔬

ACID — With Fintech Examples

▼

ACID properties ensure that database transactions are processed reliably. Critical for financial systems where partial updates can cause data corruption.

A
ATOMICITY
A transaction is all or nothing. If any step fails, the entire transaction is rolled back — no partial updates.
💡 Example: Loan disbursement of ₹5L — debit lender's account AND credit user's account. If credit fails → debit is also undone.
C
CONSISTENCY
A transaction brings the database from one valid state to another. All rules (constraints, foreign keys) must hold before and after.
💡 Example: A loan application's user_id must always reference an existing user — FK constraint ensures this.
I
ISOLATION
Concurrent transactions execute as if they were serial (one at a time). Changes of one transaction are not visible to others until committed.
💡 Example: Two agents approving the same loan simultaneously — isolation prevents double approval / race conditions.
D
DURABILITY
Once committed, a transaction persists permanently even if the system crashes immediately after.
💡 Example: Loan approval written to disk. If server crashes 1 second later, the approval record still exists when server restarts.

Transactions & Isolation Levels
💱

Transaction Commands + Isolation Levels

▼

A transaction is a sequence of SQL operations treated as a single logical unit. Transaction Control Language (TCL) commands manage their lifecycle.

BEGIN / START TRANSACTION

Marks the start of a transaction block. All subsequent statements are part of this transaction until committed or rolled back.

COMMIT

Makes all changes permanent. After COMMIT, the changes cannot be rolled back. Triggers Durability (the D in ACID).

ROLLBACK

Undoes all changes since the last BEGIN or SAVEPOINT. Restores the database to its previous consistent state.

SAVEPOINT

Creates a checkpoint within a transaction. You can ROLLBACK TO SAVEPOINT without undoing the entire transaction.

Isolation Levels control how transactions interact with each other when running concurrently:

Isolation LevelDirty ReadNon-repeatable ReadPhantom ReadPerformance
READ UNCOMMITTED✗ Possible✗ Possible✗ PossibleFastest
READ COMMITTED✓ Safe✗ Possible✗ PossibleFast
REPEATABLE READ✓ Safe✓ Safe✗ PossibleMedium
SERIALIZABLE✓ Safe✓ Safe✓ SafeSlowest
💡 Fintech note: CreditMantri would use REPEATABLE READ (MySQL default) or higher for loan processing — you don't want a user's credit score changing mid-approval-check.

All aggregates in one query Medium
COUNT, SUM, AVG, MIN, MAX on loan_applications
▼ 
SELECT
  COUNT(*) AS total_loans,
  SUM(amount) AS total_amount,
  AVG(amount) AS avg_amount,
  MIN(amount) AS min_amount,
  MAX(amount) AS max_amount
FROM loan_applications;
Explanation
A single-row summary of all loan data. Returns total count, total disbursed amount, average loan size, smallest and largest loans.
💡 Interview Tip: COUNT(*) counts all rows including NULLs. COUNT(column) counts only non-NULL values — this distinction is frequently tested!

Conditional COUNT with CASE WHEN Medium
Count approved applications without a subquery
▼ 
SELECT
  COUNT(*) AS total_applications,
  COUNT(CASE WHEN status = 'Approved' THEN 1 END) AS approved,
  COUNT(CASE WHEN status = 'Rejected' THEN 1 END) AS rejected,
  AVG(amount) AS avg_amount
FROM loan_applications;
Explanation
COUNT(CASE WHEN ... THEN 1 END) counts only rows matching the condition — the CASE returns NULL for non-matching rows and COUNT ignores NULLs.
💡 Interview Tip: This is called a "pivot" technique. Alternative: SUM(CASE WHEN status='Approved' THEN 1 ELSE 0 END) which adds 0 instead of NULL.

INNER JOIN — User loan details Easy
Combine users and loan_applications, excludes users with no loans

  u.name,

LEFT JOIN — Loan count per user (including 0) Medium
LEFT JOIN + GROUP BY, COUNT trap
▼ 
SELECT
  u.name,
  COUNT(la.application_id) AS total_applications
FROM users u
LEFT JOIN loan_applications la
  ON u.user_id = la.user_id
GROUP BY u.name;
Explanation
Users with no loans show 0 because COUNT(application_id) counts only non-NULL values. If we used COUNT(*), those users would show 1.

Users with NO loans — LEFT JOIN + IS NULL Medium
Anti-join pattern to find unmatched rows

FROM users u
LEFT JOIN loan_applications la

Loans above each lender's own average — Correlated Subquery Hard
Subquery references outer query's current row
▼ 
SELECT lender, amount
FROM loan_applications la1
WHERE amount > (
  SELECT AVG(amount)
  FROM loan_applications la2
  WHERE la2.lender = la1.lender
);
Explanation
A correlated subquery runs once per outer row. For each loan, it computes the average for that specific lender and checks if the loan exceeds it. References outer query alias la1.
💡 Interview Tip: Correlated subqueries can be slow (N+1 problem). The same result is often achievable with a window function AVG(amount) OVER (PARTITION BY lender) which is more efficient.

Window Functions
5 queries
19
ROW_NUMBER vs RANK vs DENSE_RANK Hard
The classic trio — understand the difference
▼ 
SELECT
  name,
  credit_score,
  ROW_NUMBER() OVER (ORDER BY credit_score DESC) AS row_num,
  RANK() OVER (ORDER BY credit_score DESC) AS `rank`,
  DENSE_RANK() OVER (ORDER BY credit_score DESC) AS dense_rank
FROM users;
Explanation
All three rank by credit_score descending, but handle ties differently:

• ROW_NUMBER: Always unique (1,2,3,4...) — ties get arbitrary order
• RANK: Ties get same rank, then skips numbers (1,1,3,4...)
• DENSE_RANK: Ties get same rank, no skip (1,1,2,3...)
💡 Interview Tip: If two users both have score 780, RANK gives them both rank 2 and next person gets rank 4. DENSE_RANK gives them both rank 2 and next person gets rank 3. Most frequently asked window function question.
ROW_NUMBERRANKDENSE_RANKMust Know

RANK within each lender — PARTITION BY Hard
Rank users by credit score within each lender's pool
▼ 
SELECT
  u.name,
  l.lender,
  u.credit_score,
  RANK() OVER (
    PARTITION BY l.lender
    ORDER BY u.credit_score DESC

FROM users u
JOIN loan_applications l
  ON u.user_id = l.user_id;
Explanation
PARTITION BY splits the window into groups per lender. RANK resets to 1 for each lender. Answers "highest credit-score borrower at HDFC Bank?" without complex subqueries.
💡 Interview Tip: PARTITION BY in window functions is like GROUP BY but without collapsing rows. You keep all rows AND get group-level computations alongside them.

📘 DBMS Theory
Q What is the difference between DBMS and RDBMS? ▼

A DBMS (Database Management System) is any system that manages data storage and retrieval. An RDBMS (Relational DBMS) is a specific type where data is organized in related tables using a relational model and queried via SQL.

Key differences: RDBMS enforces relationships via foreign keys, supports ACID transactions, and uses SQL as the standard query language. Examples: MySQL, PostgreSQL are RDBMS; early file-based systems like dBASE were just DBMS.

Q What is the difference between a Primary Key and a Unique Key? ▼

Primary Key: Cannot be NULL, only ONE per table, automatically creates a clustered index. Uniquely identifies every row.

Unique Key: Can have ONE NULL value (in most databases), a table can have MULTIPLE unique keys, creates a non-clustered index. Prevents duplicates but allows a single NULL.

In CreditMantri: user_id is the PK. A column like pan_number could be a Unique Key — no two users can share a PAN, but a user might not have provided it yet (NULL).

Q Explain 1NF, 2NF, 3NF with a single example. ▼

Start with a messy table: (StudentID, Subject, Teacher, TeacherPhone)

1NF: Ensure atomic values — if a student has multiple subjects, each gets its own row. No comma-separated lists in a cell.

2NF: If PK is (StudentID, Subject), Teacher depends only on Subject (partial dependency) → move Teacher to a separate Subjects table.

3NF: TeacherPhone depends on Teacher (not on the PK) — this is a transitive dependency → move TeacherPhone to a Teachers table.

Q What is Denormalization and when would you use it? ▼

Denormalization is intentionally introducing redundancy to improve read performance at the cost of some write complexity.

Use it when: your system is read-heavy (analytics, reporting), joins are too expensive, or you're building a data warehouse. For CreditMantri's loan approval dashboard, storing a pre-computed user_name in loan_applications avoids a JOIN on every dashboard load.

Trade-off: updates become slower (must update in multiple places) and risk data inconsistency. It's a deliberate, measured decision — not a design mistake.

Q What are the ACID properties? Give a real fintech example. ▼

Using a loan disbursement scenario where ₹5,00,000 is transferred from lender to borrower:

Atomicity: Both debit (lender's account) and credit (borrower's account) happen together, or neither does. No half-transfers.

Consistency: Total money before = total money after. No money is created or destroyed. All FK constraints hold.

Isolation: If two agents trigger the same disbursement simultaneously, one must wait — they can't read each other's uncommitted changes.

Durability: Once the disbursement is committed and you see "₹5L credited," that fact survives even if the server crashes 1 second later.

Q What is the difference between DELETE, TRUNCATE, and DROP? ▼

DELETE: DML command. Removes specific rows based on WHERE clause. Can be rolled back. Fires triggers. Logs each row deletion. Slow on large tables.

TRUNCATE: DDL command. Removes ALL rows instantly. Cannot be rolled back (in most databases). Does NOT fire row-level triggers. Resets AUTO_INCREMENT. Much faster than DELETE for entire table wipe.

DROP: DDL command. Removes the ENTIRE TABLE (structure + data). Cannot be rolled back. The table no longer exists after.

Memory trick: DELETE = selective surgeon, TRUNCATE = factory reset, DROP = demolition.

Q What is an index and when should you add one? ▼

An index is a separate data structure (typically a B-Tree) that the database maintains to speed up lookups on a column. Think of it as the index at the back of a book — you jump directly to the page instead of reading every page.

Add an index when: A column is frequently used in WHERE, JOIN ON, or ORDER BY clauses. The table has many rows (>10,000+). The column has high cardinality (many unique values).

Don't over-index: Indexes slow down INSERT/UPDATE/DELETE (index must be updated too). Tables with heavy writes (like CreditMantri's live loan tracking) need careful balancing. Unused indexes waste disk space and memory.

Q What is the difference between SQL and NoSQL databases? ▼

SQL (Relational): Fixed schema, ACID-compliant, great for structured/relational data, vertical scaling, uses SQL. Examples: MySQL, PostgreSQL.

NoSQL: Flexible schema, typically BASE (Basically Available, Soft state, Eventually consistent), designed for horizontal scaling, various query APIs. Examples: MongoDB, Redis, Cassandra.

Use SQL when data is structured and relationships matter (CreditMantri's core loan data). Use NoSQL when you need speed, scale, or flexibility (session tokens → Redis; event logs → Cassandra; user preferences → MongoDB).

🔍 SQL — Most Asked
Q What is the difference between WHERE and HAVING? ▼

WHERE filters individual rows before any grouping happens. It cannot reference aggregate functions.

HAVING filters groups after GROUP BY has been applied. It can reference aggregate functions.

Wrong: WHERE COUNT(*) > 2 — this will error. Correct: GROUP BY lender HAVING COUNT(*) > 2. Think of WHERE as a row-level sieve and HAVING as a group-level sieve.

Q What is the SQL query execution order? ▼

SQL clauses are written in one order but executed in a very different one:

1. FROM → 2. JOIN → 3. WHERE → 4. GROUP BY → 5. HAVING → 6. SELECT → 7. DISTINCT → 8. ORDER BY → 9. LIMIT

This explains why you can't use a SELECT alias in a WHERE clause (WHERE runs before SELECT). You CAN use an alias in ORDER BY (ORDER BY runs after SELECT). And why HAVING can use aggregates (runs after GROUP BY).

Q How do you find the Nth highest salary (or credit score)? ▼

Method 1 — DENSE_RANK (best approach):

SELECT name, credit_score FROM (SELECT name, credit_score, DENSE_RANK() OVER (ORDER BY credit_score DESC) AS rnk FROM users) t WHERE rnk = N;

Method 2 — Subquery (classic):

SELECT MIN(credit_score) FROM users WHERE credit_score IN (SELECT DISTINCT TOP N credit_score FROM users ORDER BY credit_score DESC);

DENSE_RANK is preferred because it handles ties correctly and works in all modern SQL databases.

Q What is the difference between UNION and UNION ALL? ▼

UNION: Combines results of two queries and removes duplicates. Requires an implicit DISTINCT operation — slower.

UNION ALL: Combines results keeping all duplicates. No duplicate removal — faster.

Use UNION ALL unless you specifically need deduplication. Example: combining approved loans from 2021 and 2022 — use UNION ALL since each application_id is unique anyway. Both queries must have the same number of columns with compatible data types.

Q How do you handle NULL values in SQL? ▼

NULL is not a value — it's the absence of a value. This means NULL = NULL returns UNKNOWN, not TRUE. You must use IS NULL or IS NOT NULL to check for NULLs.

Key functions:

COALESCE(col, 0) — returns first non-NULL value. Use to replace NULL with defaults.

NULLIF(a, b) — returns NULL if a = b, else returns a. Useful to avoid division-by-zero.

IFNULL(col, 'N/A') — MySQL shorthand for COALESCE with one fallback.

In CreditMantri: users 11–20 get NULL for loan columns in a LEFT JOIN. Use COALESCE(amount, 0) to show 0 instead of NULL.

Q What are window functions and how are they different from GROUP BY? ▼

GROUP BY collapses multiple rows into one row per group. You lose the detail of individual rows.

Window functions compute across a set of rows related to the current row but WITHOUT collapsing them. Each row retains its identity while gaining a group-level computation.

Example: GROUP BY lender gives one row per lender with total amount. A window function SUM(amount) OVER (PARTITION BY lender) shows each individual loan AND its lender's total in the same row — incredibly powerful for analytics like "what % of HDFC's portfolio is this loan?"

Q Write a query to find duplicate records in a table. ▼

To find duplicate names in users:

SELECT name, COUNT(*) AS cnt FROM users GROUP BY name HAVING COUNT(*) > 1;

To find the actual duplicate rows (with IDs) using ROW_NUMBER:

SELECT * FROM (SELECT *, ROW_NUMBER() OVER (PARTITION BY name ORDER BY user_id) AS rn FROM users) t WHERE rn > 1;

The second approach also lets you delete the duplicates by keeping only rn = 1 per group.

Q What is a self-join? Give an example. ▼

A self-join is a JOIN where a table is joined with itself. Useful for hierarchical or comparison scenarios within the same table.

Example — find users with a higher credit score than user 'Ravi':

SELECT u1.name, u1.credit_score FROM users u1 JOIN users u2 ON u2.name = 'Ravi' WHERE u1.credit_score > u2.credit_score;

Common use case: Employee-Manager hierarchy — employees e JOIN employees m ON e.manager_id = m.employee_id to get each employee's manager name.

Q How do you optimize a slow SQL query? ▼

Step 1 — EXPLAIN / EXPLAIN ANALYZE: Run EXPLAIN SELECT ... to see the query plan. Look for full table scans, missing indexes, high row estimates.

Step 2 — Indexing: Add indexes on columns in WHERE, JOIN, and ORDER BY. Composite indexes for multi-column filters.

Step 3 — Reduce data early: Filter with WHERE before JOINs. Use LIMIT. Avoid SELECT *.

Step 4 — Rewrite inefficient patterns: Replace correlated subqueries with JOINs or window functions. Use EXISTS instead of IN for large lists.

Step 5 — Query-level: Avoid functions on indexed columns in WHERE (WHERE YEAR(date) = 2024 prevents index use — use WHERE date BETWEEN '2024-01-01' AND '2024-12-31' instead).

Q What is the difference between a clustered and non-clustered index? ▼

Clustered Index: The physical order of rows in the table IS the index order. Only ONE per table (since data can only be physically sorted one way). The Primary Key automatically becomes the clustered index in MySQL InnoDB.

Non-clustered Index: A separate structure that points to the actual data rows. A table can have MANY non-clustered indexes. Think of it as the index at the back of a book — it doesn't change the book's page order, it just points you to the right page.

Clustered index lookups are faster for range queries. Non-clustered index lookups require an extra step to fetch the actual row (unless it's a "covering index").

Q CreditMantri scenario: Write a query to find lenders whose rejection rate increased month over month. ▼

This is a classic analytics question combining GROUP BY, conditional aggregation, and LAG:

WITH monthly_stats AS ( SELECT lender, DATE_FORMAT(application_date, '%Y-%m') AS month, SUM(CASE WHEN status='Rejected' THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS rejection_rate FROM loan_applications GROUP BY lender, month ), with_lag AS ( SELECT lender, month, rejection_rate, LAG(rejection_rate) OVER (PARTITION BY lender ORDER BY month) AS prev_rate FROM monthly_stats ) SELECT lender, month, rejection_rate, prev_rate FROM with_lag WHERE rejection_rate > prev_rate;

This pattern — CTE + window LAG + WHERE comparison — is the standard approach for period-over-period trend detection.

Q What is a VIEW and when would you use it? ▼

A VIEW is a named, saved SQL query that behaves like a virtual table. The underlying data is not duplicated — the view's SQL runs every time you query the view.

Use views when: You have a complex JOIN/aggregate query used repeatedly. You want to restrict which columns/rows different users can see (security). You want to simplify reporting queries for non-technical stakeholders.

CreditMantri example: CREATE VIEW approved_users AS SELECT u.name, u.credit_score, la.lender, la.amount FROM users u JOIN loan_applications la ON u.user_id = la.user_id WHERE la.status = 'Approved';

Now analysts can just query SELECT * FROM approved_users without knowing the JOIN.

Q A user's loan application is stuck in 'Pending'. How would you debug using SQL? ▼

Approach this systematically — first understand the data, then investigate anomalies:

Step 1 — Verify the record exists: SELECT * FROM loan_applications WHERE user_id = X AND status = 'Pending';

Step 2 — Check the user's profile: SELECT * FROM users WHERE user_id = X; — look at credit_score (if <600, might explain auto-rejection issues).

Step 3 — Check for duplicates: SELECT COUNT(*) FROM loan_applications WHERE user_id = X GROUP BY lender HAVING COUNT(*) > 1;

Step 4 — Compare with successfully approved applications of similar users to identify what's different.

This shows interviewers you think in a structured, investigative way — not just write random queries.

Q Design a table schema to track loan EMI payments for CreditMantri. ▼

Think about entities and relationships before writing DDL:

CREATE TABLE emi_payments ( payment_id INT PRIMARY KEY AUTO_INCREMENT, application_id INT NOT NULL, emi_number INT NOT NULL, due_date DATE NOT NULL, paid_date DATE, amount_due DECIMAL(12,2) NOT NULL, amount_paid DECIMAL(12,2) DEFAULT 0, status ENUM('Pending','Paid','Overdue') DEFAULT 'Pending', FOREIGN KEY (application_id) REFERENCES loan_applications(application_id) );

Key decisions: Use DECIMAL for money (never FLOAT — floating point errors in finance are catastrophic). Separate amount_due and amount_paid for partial payments. ENUM for status to enforce valid values. FK to loan_applications to maintain referential integrity.

Q What causes a Cartesian product and how do you prevent it? ▼

A Cartesian product occurs when you JOIN two tables without a join condition, producing every combination of rows. 20 users × 25 loans = 500 rows — almost always unintentional.

Common causes: Missing ON clause in a JOIN. Using a comma join instead of explicit JOIN syntax: FROM users, loan_applications with no WHERE — this is a cross join!

Prevention: Always use explicit JOIN syntax with an ON condition. Enable query reviews in production. Use EXPLAIN to spot unexpectedly large row counts in the plan.

When Cartesian products ARE intentional: CROSS JOIN to generate combinations — e.g., all possible (user, lender) pairs for a lead generation matrix.

Q What is a deadlock in a database and how can it be avoided? ▼

A deadlock occurs when two transactions each hold a lock that the other transaction needs — they wait for each other forever.

Example: Transaction A locks user_id=1 and wants loan_id=101. Transaction B locks loan_id=101 and wants user_id=1. Neither can proceed.

Prevention strategies: Always acquire locks in the same order across all transactions. Keep transactions short — minimize the time locks are held. Use lower isolation levels where business requirements allow. Set lock timeout so deadlocked transactions are automatically killed and retried.

MySQL InnoDB automatically detects deadlocks and rolls back one transaction (the "victim") to let the other proceed.
